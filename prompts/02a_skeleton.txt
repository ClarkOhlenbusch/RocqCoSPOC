**Translation rules** (follow the guide above):
- `PROVE name: signature` -> do not emit the lemma header; only emit the tactic body that would go between `Proof.` and `Qed.`
- `ASSUME x : T` -> `intros x.` unless the current proof source already imports the Angelito tactics library
- `INDUCTION n` -> `induction n.` with bullet or brace blocks for the cases
- `APPLY theorem SPLIT INTO: (1)... (2)...` -> `apply theorem.` followed by one `admit.` leaf per subgoal
- `PROVE BASE_CASE:` / `PROVE INDUCTIVE_CASE:` -> bullet or brace blocks
- `FACT h: stmt [BY ...]` -> usually `assert (h : stmt).` followed by a brace block containing only `admit.`
- `FACT h: exists ...` followed by later `ASSUME` bindings -> preserve that structure as `assert (h : exists ...).` then `destruct h as [...] .`
- `SIMPLIFY ...`, `THEREFORE ...`, and major intermediate equalities -> preserve them as `assert (...) .` blocks when they are key checkpoints in the direct proof
- Leave proofs of those assertions as standalone `admit.` leaves inside braces

**Hard requirements:**
1. Output only the tactic lines between `Proof.` and `Qed.`. Do not include `Proof.`, `Qed.`, `Admitted.`, the theorem statement, or any prose.
2. Every leaf goal must be `admit.`
3. The skeleton must compile when wrapped in `Proof. ... Admitted.`
4. Use bullets (`-`, `+`, `*`) or braces (`{ }`) to structure subgoals.
5. Every non-empty line must be valid Rocq syntax.
6. Angelito keywords are not valid Rocq syntax. Never emit uppercase Angelito keywords such as `ASSUME`, `FACT`, `SIMPLIFY`, `THEREFORE`, or `CONCLUDE`.
7. If the proof uses induction on `n`, the output should normally look like:
   `intros ... .`
   `induction n.`
   `- admit.`

8. Do not invent proof structure that is not present in the Angelito proof. But do preserve useful direct-proof structure from Angelito `FACT`, `SIMPLIFY`, and `THEREFORE` steps as Rocq `assert` / `destruct` scaffolding instead of collapsing everything to a single final `admit.`
9. Do not partially solve goals. Once a branch reaches `admit.`, stop writing tactics for that branch.
10. If the formal statement begins with `forall` binders or implications, the skeleton must introduce them first with `intros ... .` before using `induction`, `destruct`, or other goal-opening tactics.
11. After `intros n` and `induction n`, the inductive subgoal is usually `S n + 0 = S n` (or similar) with `IHn : n + 0 = n` already in context. Do not emit `pick` or `intros` for a second parameter name that induction already scoped; use `admit.` for the leaf and let the fill step work on the real goal.
12. If the Angelito text contains `PROVE INDUCTIVE_CASE:` followed by `ASSUME m : ...`, treat that `ASSUME` as naming the predecessor variable already introduced by `induction`. Do not translate it into `pick m : ...` or `intros m.` unless the actual Rocq goal in that branch visibly starts with a `forall` or implication binder.
13. Preserve bullet and brace structure exactly. If you open branches with `induction`, `destruct`, or `apply ...` that creates multiple subgoals, put each branch on its own bullet or brace-delimited block.

16. It is acceptable for the skeleton to have many `admit.` placeholders if that preserves the proof plan faithfully.
17. Every asserted proposition must itself be valid Rocq syntax. Do not copy pseudo-mathematical notation such as `card {a | ...}`, `sum_{j=...}`, `∑`, or set-builder braces from Angelito into the Rocq skeleton.
18. If an Angelito checkpoint uses notation that is not directly expressible with the visible imports, preserve the proof plan at a coarser but still valid Rocq checkpoint instead of inventing unsupported syntax.
